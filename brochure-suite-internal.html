Company thesis

Sentra is an Enterprise Software Builder. We do not write enterprise software from scratch. We don't sell SaaS subscriptions. We take Odoo Community as the universal business platform, layer Sentra's AI + automation stack on top, brand it as the customer's own system, and deploy it on AWS — turning what used to be a 6–12 month custom build into a 4–6 week branded delivery.

VPC layout (every client deploy)

04 · Atlas

Atlas — Deployment Runbook.

ATLASRUNBOOK v1.0

Standard sequence for deploying Atlas to a new client AWS account. Estimated time end-to-end (excluding data migration): 2 working days for Small/Medium tier.

Step 1 — Fork the master

# Clone Sentra master, push as new client repo
gh repo clone sentra/master-atlas
gh repo create sentra/client-{slug}-atlas --private --push --source=master-atlas
cd client-{slug}-atlas
cp .env.example .env  # fill in client-specific values

Step 2 — Provision AWS infra (CDK)

  1. Switch CDK profile to client account: aws sso login --profile client-{slug}
  2. Edit cdk/config.ts — client slug, region, tier (small/medium/enterprise), domain.
  3. cdk deploy SentraAtlasStack-{slug} — provisions VPC, subnets, RDS, EC2, ALB, S3, security groups, Route53.
  4. Verify outputs: ALB DNS, RDS endpoint, S3 bucket names — record in client onboarding doc.

Step 3 — Deploy Docker stack

ssh ec2-user@{atlas-ec2-ip}
git clone git@github.com:sentra/client-{slug}-atlas.git
cd client-{slug}-atlas
sudo docker compose up -d
sudo docker compose logs -f odoo  # wait for "Modules loaded"

Step 4 — SSL + DNS

  1. Point client DNS: atlas.{client-domain} → ALB DNS (CNAME).
  2. Issue Let's Encrypt cert: sudo certbot --nginx -d atlas.{client-domain}
  3. Verify HTTPS: curl -I https://atlas.{client-domain} returns 200 OK.

Step 5 — Brand kit + initial config

  1. In Odoo as admin: Settings → Sentra Brand Kit → upload logo (SVG + PNG fallback), set primary/accent colors, set company info.
  2. Verify: login screen, email templates, invoice PDF, dashboard top-bar all show client branding.
  3. Configure SES sender + DKIM in Settings → Email Servers. Verify with test send.
  4. Configure WhatsApp Business API credentials in Settings → Sentra WhatsApp. Send test message to internal number.

Step 6 — Data migration

  • Customers / suppliers: CSV import via standard Odoo import wizard.
  • Products: CSV with columns name, default_code, list_price, qty_available, categ_id.
  • Open invoices: manual import or one-shot script in scripts/import-open-invoices.py.
  • Historical accounting: opening balance journal entry only (don't re-import old transactions).

Step 7 — Smoke tests (must pass before handover)

  1. Create a test customer + sales order + invoice. Confirm WhatsApp confirmation sent.
  2. Mark invoice paid via Stripe test mode. Confirm n8n flow updates Odoo + sends receipt.
  3. Reduce a product stock to below reorder point. Confirm low-stock-alert fires.
  4. Run /health endpoint — all services green.
  5. Trigger nightly backup manually (aws rds create-db-snapshot). Confirm S3 backup arrives.

Step 8 — Handover checklist

  • Admin credentials transferred via 1Password vault share.
  • 2× training sessions for client admin + power users (recorded).
  • Runbook URL + on-call rotation shared.
  • Monthly support contract signed and Stripe subscription active.
  • 30-day post-launch check-in scheduled.
NEVER Hard-code credentials in the repo. Always read from AWS Secrets Manager via the sentra_brand_kit/utils/secrets.py helper. One leak ends the company.

AI ticket responder — standard config

  • Routing: incoming message hits Chatwoot → n8n webhook fires → n8n calls Cortex with conversation history + customer record from Atlas → Cortex returns suggested reply + confidence score.
  • Confidence >0.85: auto-reply sent. Logged with "Sent by AI" tag.
  • Confidence 0.5–0.85: draft reply pre-filled in agent inbox. Agent reviews + sends in one click.
  • Confidence <0.5: escalated to human queue. Cortex still suggests likely intent + relevant docs in sidebar.
  • Guardrails: never auto-quote prices, never auto-confirm refunds >$X, never reply to anything tagged legal or complaint.
  • Knowledge source: always Cortex (the client's private brain). Never general internet search. PII redaction on by default.

Echo — Deployment Runbook.

ECHORUNBOOK v1.0

Standard sequence for Echo deployment. Most steps are common with Atlas; only Echo-specific actions are listed. Estimated time end-to-end: 1.5 working days + WhatsApp Business API approval lag (3–7 days).

Step 1 — Fork & provision (same as Atlas)

gh repo create sentra/client-{slug}-echo --private --source=master-echo
cd cdk && cdk deploy SentraEchoStack-{slug}

Step 2 — Chatwoot stack up

ssh ec2-user@{echo-ec2-ip}
git pull
sudo docker compose up -d   # brings up: chatwoot-rails, chatwoot-sidekiq, postgres, redis, n8n
# Run rails migration on first boot
sudo docker compose exec rails bundle exec rails db:chatwoot_prepare

Step 3 — WhatsApp Business API setup (Meta direct)

  1. Client creates Meta Business Manager account (we cannot do this for them — legal owner).
  2. Client adds Sentra as a partner with manage_whatsapp_settings permission.
  3. Verify business via Meta's verification process (5–7 days — allow buffer in timeline).
  4. Apply for WhatsApp Business API access. Use template request boilerplate in docs/wa-bm-application.md.
  5. Once approved: register phone number (must NOT be already on consumer WhatsApp).
  6. Generate permanent System User token. Store in Secrets Manager /sentra/{slug}/echo/wa-token.
  7. Configure webhook in Meta dev console → https://echo.{client-domain}/webhooks/whatsapp.
  8. Create at least 3 message templates (welcome, order-confirm, support-handoff). Submit for Meta approval (24–48hrs).

Step 4 — Channel wire-up in Chatwoot

  • Settings → Inboxes → New — one inbox per channel (WhatsApp, Email, Web Widget, etc.).
  • Email inbox → configure SES inbound rule to forward to support+{inbox-id}@{client-domain}.
  • Web widget → copy embed snippet to client website team.

Step 5 — AI responder activation

  1. In n8n, import flow echo-ai-responder.json from sentra/master-flows.
  2. Set Cortex endpoint URL + API key (from Cortex deploy on adjacent EC2 / VPC peer).
  3. Set confidence thresholds in flow settings (defaults: 0.85 / 0.5).
  4. Add guardrail rules: refund cap, legal-tag block, language whitelist.
  5. Activate flow. Test by sending message from internal WhatsApp number.

Step 6 — Atlas + Cortex integration

  • Configure Chatwoot → Atlas customer sync via sentra_n8n_bridge: every new contact in Chatwoot creates/links a res.partner in Atlas Odoo.
  • Configure Cortex — ingest Chatwoot conversation history nightly into Cortex vector store. Helps AI replies stay on-brand.

Step 7 — Smoke tests

  1. Send WhatsApp to client number from your phone → conversation appears in Chatwoot.
  2. AI responder either auto-replies (if confident) or drafts reply.
  3. Reply manually → arrives back on your WhatsApp within 5 seconds.
  4. Send email to support@{client-domain} → appears as new conversation.
  5. Verify Atlas res.partner created/linked. Verify dashboard counts increment.

Cortex — Product Specification.

CORTEXCOMPANY BRAIN

Cortex is Sentra's private AI brain product. Every client gets a fully isolated LLM stack trained / grounded on their own data. The chat interface is OpenWebUI rebuilt with the client's branding (OpenWebUI's branding is fully stripped — see runbook). Cortex powers the AI assistants in Atlas and Echo, the question-answer chat, and the daily executive summary in Lens.

Component stack

LayerComponentWhy
LLM runtimeOllamaSelf-hosts open weights on EC2 GPU. No data leaves the client VPC.
Default modelsLlama 3.1 8B (chat), Mistral 7B (summarisation)Best quality/cost balance on g5.xlarge.
Heavy-context modelLlama 3.1 70BOptional. Requires g5.4xlarge or g6.2xlarge minimum.
Specialist modelsMixtral 8x7B, Qwen 2.5 / Qwen 3, DeepSeek, DeepSeek CoderLoaded on demand. Coder for client-side code generation tasks.
Hybrid (optional)Anthropic Claude API / OpenAIFor clients with no PII concern + premium quality requirement. Cost is passed through.
Vector DBQdrantFaster + simpler ops than pgvector at scale. Per-client isolated collection.
RAG orchestrationLangChain (Python)Standard. Custom chains live in sentra/master-cortex/chains/.
Chat UIOpenWebUI (white-labeled fork)Mature, theme-able. Sentra fork removes upstream branding.
Docs / wiki baseOutlineModern UI, markdown-native, easy SSO. Where the human-curated knowledge lives.
Embeddingsnomic-embed-text via OllamaLocal, free, decent quality. Switch to BAAI/bge-m3 for multilingual.
RerankerBAAI/bge-reranker-v2-m3Boosts answer relevance ~15%. Optional but recommended.

Document connectors (ingestion sources)

  • Outline pages — native ingest via Outline API. Re-indexed nightly via n8n.
  • Google Drive — service account read access, scoped to specific folders.
  • SharePoint / OneDrive — Microsoft Graph API, app registration in client tenant.
  • Slack — bot token, ingests selected channels.
  • S3 — bucket of PDFs, docs, slides. Walked daily.
  • Atlas (Odoo) records — via the standard sentra_n8n_bridge. SOPs, customer notes, invoices.
  • Echo conversations — resolved tickets ingested nightly to improve responder accuracy.

RAG pipeline (default)

  1. Chunking: 512 tokens with 64-token overlap, semantic-aware splitter for markdown / PDFs.
  2. Embedding: nomic-embed-text (768-dim) per chunk, stored in Qdrant.
  3. Retrieval: top-20 by cosine similarity → rerank to top-5 by bge-reranker.
  4. Generation: top-5 chunks + question + system prompt → Llama 3.1 8B.
  5. Citation: every answer returns source chunk IDs → rendered as clickable refs in OpenWebUI.
  6. PII redaction: Microsoft Presidio runs on every chunk before embedding. Configurable allow-list per client.

Inference cost calculator (rule of thumb)

  • Self-hosted on g5.xlarge: ~$480/mo fixed regardless of usage (1xA10G GPU). Effective tokens are essentially free at this point.
  • Hybrid Claude/OpenAI usage: price per million tokens varies; client pays AWS Bedrock or vendor list + 10%.
  • Heavy summarisation jobs (e.g. weekly board report): budget ~$2–$5/run on hosted models. Negligible on self-hosted.

cdk deploy SentraCortexStack-{slug}

ssh ec2-user@{cortex-gpu-ip}
curl -fsSL https://ollama.com/install.sh | sh && sudo systemctl enable --now ollama

# Standard set (always)
for m in llama3.1:8b mistral:7b nomic-embed-text; do ollama pull $m; done

# Optional, pull on demand: mixtral:8x7b, qwen2.5:14b, deepseek-coder:6.7b, deepseek-r1:7b
ollama list && curl http://localhost:11434/api/tags  # verify

KEY=$(aws secretsmanager get-secret-value --secret-id /sentra/{slug}/cortex/qdrant-key --query SecretString --output text)
sudo docker run -d --name qdrant -p 6333:6333 -v /data/qdrant:/qdrant/storage \
  -e QDRANT__SERVICE__API_KEY=$KEY qdrant/qdrant:latest

Step 4 — OpenWebUI deploy + branding strip

  1. Clone Sentra fork: git clone git@github.com:sentra/openwebui-whitelabel
  2. Apply client brand kit: ./scripts/apply-brand.sh {slug} — this script:
    • Replaces /static/favicon.png, /static/splash.png, /static/logo*.svg with client assets.
    • Patches app/manifest.json name/short_name.
    • Patches app/static/i18n/en/translation.json "Open WebUI" → client product name (e.g. "Atlas Brain" or whatever the client called it).
    • Patches tailwind.config.js color palette to client primary/accent.
    • Replaces footer "Open WebUI" link → client support URL.
  3. Build & run: docker compose up -d openwebui.
  4. Configure backend in OpenWebUI: Settings → Connections → add Ollama URL http://{cortex-gpu-ip}:11434.
  5. Visual QA: load login screen, chat screen, settings — zero "Open WebUI" strings should remain. Run grep -r "Open WebUI" /app to confirm.

Step 5 — Outline (wiki) deploy

cd /opt/outline
docker compose up -d
# Configure SSO via Authentik. Apply Sentra brand kit (logo, primary color).

Step 6 — Initial ingestion

  1. Configure data sources in cortex/config/sources.yaml — one block per connector (Outline, Drive, SharePoint, Slack, S3, Atlas).
  2. Run initial ingestion: python -m cortex.ingest --all --client {slug}. Expect ~30 min–2 hours depending on document volume.
  3. Verify Qdrant collection count: curl -H "api-key: $KEY" http://localhost:6333/collections/{slug}.
  4. Schedule nightly re-index: n8n flow cortex-reindex, cron 0 2 * * *.

Step 7 — Atlas integration (sidebar widget)

  • Install sentra_cortex_assistant module in Atlas Odoo (already in master).
  • Configure Cortex API URL + API key in Settings → Sentra Cortex.
  • Verify: open any record in Odoo → Cortex sidebar appears → ask "summarise this customer's history" → answer with citations.

Step 8 — Smoke tests & cost monitor

  1. Ask 5 questions covering: a customer policy, a SOP, a recent invoice, a Slack thread, a Drive PDF. All must return correct answers with citations.

Lens — Product Specification.

LENSEXEC ANALYTICS

Lens is the Sentra executive analytics product. Default base is Metabase (best UX for non-technical execs). For clients with analyst-heavy teams who want notebook-style exploration, swap to Apache Superset. Lens always pairs with Atlas (it reads from the Atlas Postgres replica). It can also pair with Echo and Cortex telemetry.

When to use which base

Metabase (default)Execs who want a clean dashboard and the occasional "ask a question" feature. SMEs to mid-market.

Data plumbing

  • Source: Atlas Postgres (Odoo). We create a read replica in RDS so Metabase queries don't impact production.
  • Materialised views: for any dashboard query taking >3s, create a materialised view refreshed via n8n cron. Lives in schema sentra_lens.
  • Echo data: Chatwoot replica → joined into Customer Health dashboard.
  • Cortex telemetry: usage stats, popular questions, knowledge gaps → "Brain Insights" board.

AI alert engine

The Lens differentiator vs vanilla Metabase. Implemented entirely in n8n.

  • Every scheduled metric query runs through n8n lens-anomaly-detector flow.
  • Compares current value vs 30-day rolling baseline + 2σ band.
  • Anomaly → sent to Cortex with context. Cortex generates a 2-sentence English explanation.
  • Delivered via WhatsApp to the metric's owner. Severity tag (info / warning / urgent) in the message.
  • Daily 07:00: lens-daily-briefing flow — Cortex summarises last 24h across all dashboards into a 60-second exec voice note (Vapi optional) + WhatsApp text.

Embedding into Atlas

  • Module sentra_lens_embed iframes the relevant Metabase dashboard inside Odoo screens.
  • Auth: signed JWT (1-hour expiry) — user logs into Odoo, dashboard appears already authenticated.
  • Examples: Sales pipeline screen → Sales Pulse panel; Project page → Project KPIs; Helpdesk page (Echo) → Customer Health.

Lens — Deployment Runbook.

LENSRUNBOOK v1.0

Lens is the lightest of the four products to deploy. Estimated time: 1 working day, plus dashboard tuning per client (1–2 days).

Step 1 — Provision read replica

# Add a read replica to Atlas RDS
aws rds create-db-instance-read-replica \
  --db-instance-identifier sentra-{slug}-atlas-replica \
  --source-db-instance-identifier sentra-{slug}-atlas-primary \
  --db-instance-class db.t3.medium

Step 2 — Metabase up

ssh ec2-user@{lens-ec2-ip}
git clone git@github.com:sentra/client-{slug}-lens.git
cd client-{slug}-lens
sudo docker compose up -d  # brings up: metabase, postgres (metabase metadata)
# wait for metabase to finish init — check logs for "Metabase started"

Step 3 — Connect data source

  1. In Metabase admin: Data → Add database → PostgreSQL.
  2. Host: replica endpoint. Database: {slug}_odoo. User: lens_readonly (created by CDK).
  3. Verify connection. Sync schema (~5 min).

Step 4 — Import standard dashboards

# Sentra dashboards are versioned as Metabase serialised export.
docker exec metabase java -jar /app/metabase.jar load \
  /sentra-dashboards/v1 --on-error continue
# Then re-map each card's database ID to the new connection.
python scripts/remap-database-ids.py --client {slug}

Step 5 — Brand kit

  1. Apply Sentra Metabase whitelabel patch (Enterprise edition feature, but achievable in Community via custom CSS overlay):
    • Inject /app/dist/css/sentra-brand-{slug}.css via reverse proxy.
    • Replace logo, favicon, login background.
    • Hide the "Powered by Metabase" footer via CSS display: none.
  2. Verify: login screen, dashboard chrome, embedded view all show client branding.

Step 6 — AI alert flows

  1. In n8n, import flows lens-anomaly-detector.json and lens-daily-briefing.json.
  2. Configure metric watch list with the client during discovery (which 5–10 numbers does the CEO want pinged about?).
  3. Configure WhatsApp recipients per metric (owner mapping).
  4. Test by manually injecting an outlier into a test metric → alert should arrive on phone within 60 seconds.

Step 7 — Embed into Atlas

  • In Atlas: Settings → Sentra Lens → paste Metabase URL + secret signing key.
  • Configure embed mappings (which Odoo screen shows which dashboard panel).
  • Verify: open Sales pipeline in Odoo → Sales Pulse panel renders embedded.

Step 8 — CEO cockpit configuration

This is the white-glove step. Sit with the CEO for 30 min, ask "what 5 numbers do you want on your phone every morning?" — then build exactly that dashboard, no more, no less. Set the AI daily briefing to summarise it in 60 seconds.

Smoke tests

  1. All 7 standard dashboards render in <3 seconds.
  2. One metric anomaly fires correctly to test WhatsApp number.
  3. Daily briefing arrives on schedule with correct content.
  4. Embedded view in Atlas shows the right dashboard for the right user role.

Discovery questions (ask all of these on call 1)

  1. How many people on the team? Across how many locations / entities?
  2. Which 3 tools / spreadsheets does your team complain about most?
  3. How do customers reach you today? Which channels? What % is unanswered >24h?
  4. What's your current monthly software bill (everything — CRM, accounting, support, BI, AI tools)?
  5. If we built one platform that replaced 80% of those tools and gave you AI on top, what would that be worth annually?
  6. Who's the executive sponsor? Who'll use it daily? Who's the technical owner on your side?
  7. Do you have a preferred AWS account or do you want us to host?
  8. What's the deadline driving this? (Always probe for the real why-now.)
  9. Have you tried to fix this before? What happened? (Important — previous failures = scars.)
  10. What does success look like to you 6 months after go-live?
